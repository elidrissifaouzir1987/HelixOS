# Data Model: Durable One-Shot Dispatch

This model implements the no-effect R1 boundary in [spec.md](spec.md). The PLAN-004
coordinator V1 schema and retained history remain authoritative. Its base operation stays
`PREPARING` with a held reservation while dispatch can still execute or be ambiguous;
only exact final no-consumption reconciliation may append the existing base
`PREPARING -> FAILED` and release that hold once. PLAN-005 adds an effective dispatch
overlay, a separate adapter inbox store and signed cross-domain evidence. All integers
are non-negative safe integers at or below `2^53 - 1`; all identifiers are
length/character bounded; all digests are fixed SHA-256 values.

## 1. Type conventions

- Wire contracts use strict canonical JSON with unknown fields denied.
- Signature profiles use Ed25519 and distinct domain-separated messages.
- Positive authority types are private/non-constructible outside their owner and are
  not serializable unless explicitly described as a wire contract.
- Untrusted requests carry lookup keys and expected digests only.
- UTC is trusted wall-clock evidence; deadlines are exclusive boot-bound monotonic
  values. Both are required and must agree with the retained PLAN-004 context.
- Public logs, `Debug`, metrics and audit projections omit internal identifiers/digests;
  canonical sovereign store rows retain the required exact values.

## 2. Portable request and authority entities

### 2.1 `DispatchLookupRequestV1`

Untrusted bounded input used only to locate and compare authoritative state.

| Field | Rule |
|---|---|
| `contract_version` | exactly v1 |
| `operation_id` | bounded identifier; lookup only |
| `expected_plan_id` | exact PLAN-001 digest |
| `expected_preparation_attempt_id` | exact PLAN-004 attempt digest |
| `expected_preparation_transition_generation` | non-zero safe integer |
| `caller_deadline_monotonic_ms` | exclusive, current boot only |

It carries no positive operation, lease, approval, recovery, budget or dispatch
authority.

### 2.2 `DispatchAttemptIdV1`

Random domain-separated 256-bit identity generated by coordinator-owned entropy after a
valid preliminary reload. It is never caller-supplied or reused. It binds uncertain
commit/readback to one exact attempt and differs from grant ID and one-shot nonce.

### 2.3 `ReadyDispatchContextV1`

Private coordinator projection created from one invariant-valid current V2 view. It
contains:

- lookup/plan/preparation/operation identities and current generations;
- canonical effect descriptor and exact PLAN-001 plan custody;
- replay, reservation, recovery and preparation-event bindings;
- current lease/authorization/workload/trust/policy/catalog/capability generations and
  digests from injected trusted views;
- destination adapter/protocol/capability binding;
- boot, instance and supervisor epochs;
- trusted UTC/monotonic samples and the computed exclusive grant deadline;
- preliminary store/root/schema/generation summary.

It is non-wire, non-`Clone`, non-Serde, redacted in `Debug`, and cannot itself authorize
delivery.

### 2.4 `DispatchGuardSetV1`

Short-lived guards acquired in the PLAN-004 global order. Each carries an owner,
generation and revocation observation. The set covers supervisor admission, trust,
workload, lease, authorization, policy, catalog, capability, replay, operation,
reservation, recovery, destination/protocol and signing authority. Release order is the
reverse of acquisition.

### 2.5 `DispatchCommitPermitV1`

Non-cloneable linearizable supervisor permit retained across final comparison and the
coordinator V2 transaction. Its deadline is the earlier of caller/grant authority and
the bounded permit ceiling defined by the implementation contract. PAUSE/HALT or owner
loss revokes it and prevents any new grant commit.

## 3. Signed execution grant

### 3.1 `ExecutionGrantProtectedV1`

Canonical protected payload signed under
`HELIXOS\0EXECUTION-GRANT\0V1\0`.

| Group | Required values |
|---|---|
| Contract | schema, digest and signature profile, grant signer key/purpose |
| One-shot identity | grant ID, dispatch attempt ID, nonce |
| Operation | operation ID, state generation, preparation attempt/transition, plan ID |
| Request authority | task/workload, lease and authorization generations/digests |
| Decision authority | trust, policy, catalog and capability generations/digests |
| Effect descriptor | typed intent, portable target, precondition/content digests and bounded lengths/media type; never replacement bytes |
| Budget/recovery | reservation ID/digest/vector binding and recovery reference/profile digest |
| Destination | adapter identity, protocol version and adapter capability digest |
| Time/fencing | boot ID, instance/supervisor epochs, trusted UTC/monotonic issue samples, exclusive deadline |

Validation requires every binding to match the retained final comparison. The exclusive
deadline is `min(all authority deadlines, caller deadline, issue_monotonic + 5000 ms)`.

### 3.2 `SignedExecutionGrantV1`

| Field | Rule |
|---|---|
| `protected` | validated `ExecutionGrantProtectedV1` |
| `grant_digest` | SHA-256 of canonical protected bytes |
| `signature` | canonical base64url Ed25519 signature |

The complete canonical envelope bytes are retained. Grant ID, operation ID and nonce are
each create-only unique in coordinator and adapter domains.

### 3.3 `AuthenticExecutionGrantV1`

Non-serializable result of strict canonical decode, digest recomputation, key-purpose /
trust / revocation lookup and signature verification. It carries the verified public-key
fingerprint and cannot be constructed from deserialized fields alone.

## 4. Signed execution receipt

### 4.1 `ExecutionReceiptDecisionV1`

Closed enum:

- `CONSUMED`: exact grant authority durably consumed once; the only positive decision.
- `REFUSED_DEFINITE`: no consumption occurred and a permanent tombstone plus guarded
  no-in-flight proof exists. The closed post-`RECEIVED` refusal reason is exactly
  `GRANT_EXPIRED`, `SUPERVISOR_EPOCH_MISMATCH` or `ADAPTER_PAUSED`.

Malformed/forged/unsupported/conflicting inputs never produce a positive receipt; they
create local diagnostic or quarantine evidence.

`DESTINATION_MISMATCH`, `PROTOCOL_UNSUPPORTED`, `CAPABILITY_MISMATCH` and
`INBOX_CAPACITY_EXHAUSTED` are rejected before `RECEIVED`. They create durable local
diagnostic or quarantine evidence, never a signed receipt, and cannot prove
no-consumption or authorize reservation release.

### 4.2 `ExecutionReceiptProtectedV1`

Canonical payload signed by the destination adapter under
`HELIXOS\0EXECUTION-RECEIPT\0V1\0`.

| Group | Required values |
|---|---|
| Contract | receipt schema/profile and adapter receipt signer key/purpose |
| Identity | receipt ID, grant ID/digest, operation, destination adapter |
| Inbox | adapter root ID, inbox generation, consumption/refusal generation |
| Fencing | independently observed boot/supervisor epoch and observer generation |
| Decision | closed decision and optional closed refusal code |
| Time/trace | trusted UTC/monotonic sample and bounded opaque trace ID |

### 4.3 `SignedExecutionReceiptV1` and `AuthenticExecutionReceiptV1`

The signed envelope mirrors grant digest/signature rules with the receipt domain and
adapter trust profile. The authentic type is non-wire and carries verified public-key
fingerprint. Grant-signing keys cannot verify receipts; receipt keys cannot verify
grants or plans.

## 5. Coordinator V2 persistence overlay

All PLAN-004 V1 tables remain byte-semantically unchanged. `user_version=2` indicates
that the reviewed dispatch overlay is complete. Effective lifecycle is derived from the
overlay while the base prepared row remains `PREPARING` and its reservation remains
`HELD`.

### 5.1 `DispatchStoreMetadataV1`

Singleton dispatch extension identity:

- extension/application/schema/format versions and schema digest;
- root binding to the unchanged coordinator V1 identity;
- dispatch, delivery, receipt, reconciliation, event and migration generations;
- queue capacity and control-lane profile;
- root lifecycle `ACTIVE` or `RESTORE_PENDING`;
- optional restore/index/attestation digests.

Every generation is monotonic, non-reusable and no greater than the dispatch store
generation.

### 5.2 `CoordinatorV2MigrationReceiptV1`

Permanent one-row receipt binding migration attempt, source V1 schema/root/summary,
verified backup digest, target overlay schema digest, trusted time, result generation
and tool/source identity. `user_version=2` is written last in the same transaction.

### 5.3 `DispatchComparisonRecordV1`

Exact final capture persisted for one attempt:

- all PLAN-004 current authority generations/digests;
- operation/preparation/replay/reservation/recovery/event bindings;
- destination/protocol/signing key profile;
- trusted time/fencing samples;
- preliminary/final context digests and comparison result.

Only equality across every required group permits grant commit.

### 5.4 `DispatchGrantRecordV1`

Create-only row keyed by grant ID and additionally unique by operation ID and nonce:

- dispatch attempt and operation/preparation bindings;
- exact canonical signed grant bytes and digest;
- signer purpose/key/profile/fingerprint;
- issue/deadline and destination/protocol;
- created generation and current custody state.

Bytes are immutable and are the only deliverable grant representation.

### 5.5 `DispatchRecordV1`

Current effective overlay for one operation:

| Field | Rule |
|---|---|
| `operation_id` | exactly one base PLAN-004 operation |
| `grant_id` / `dispatch_attempt_id` | immutable create-only bindings |
| `state` | `DISPATCHING`, `EXECUTING`, `OUTCOME_UNKNOWN`, `RECONCILIATION_REQUIRED` or `FAILED` |
| `state_generation` | latest permanent overlay transition |
| `current_event_id` | exact event for latest state |
| `receipt_id` | required for `EXECUTING` and definite-refusal `FAILED`; optional retained evidence in reconciliation |
| `reconciliation_id` | required for `RECONCILIATION_REQUIRED` and `FAILED` |

No delete, ID replacement or backward transition exists.

### 5.6 `DispatchTransitionV1`

Permanent append-only transitions:

```text
BASE PREPARING -> DISPATCHING
DISPATCHING -> EXECUTING
DISPATCHING -> OUTCOME_UNKNOWN
OUTCOME_UNKNOWN -> RECONCILIATION_REQUIRED
RECONCILIATION_REQUIRED -> FAILED   # exact no-consumption proof only
```

`REFUSED_DEFINITE` appends the complete
`DISPATCHING -> OUTCOME_UNKNOWN -> RECONCILIATION_REQUIRED -> FAILED` overlay history,
the signed receipt and reconciliation record. The same V2 transaction appends the
existing PLAN-004 base `PREPARING -> FAILED`, releases the exact held reservation once
and appends its base event. A `CONSUMED` receipt discovered after unknown custody is
retained in `RECONCILIATION_REQUIRED`; it cannot jump to `EXECUTING` without a later
effect-aware contract.

### 5.7 `DispatchOutboxV1`

One restricted deliverable row per grant:

- exact grant record reference/digest;
- delivery state `PENDING`, `HANDED_OFF`, `ACKNOWLEDGED`, `QUIESCED` or `UNKNOWN`;
- delivery attempt/generation, handoff-guard identity and deadline;
- response/receipt reference when known.

It never stores reconstructed bytes and is written in the initial dispatch transaction.

### 5.8 `DispatchDeliveryAttemptV1`

Append-only attempt evidence for exact redelivery:

- attempt number and original grant reference;
- pre-handoff pause/deadline/fencing observation;
- linearizable handoff guard/generation;
- confirmed no-send, possible handoff or acknowledgement classification;
- adapter root/epoch and readback generation where observed.

No attempt changes grant ID, nonce, bytes or deadline.

### 5.9 `CoordinatorDispatchReceiptV1`

Exact canonical signed receipt bytes, digest, verified adapter public-key fingerprint,
grant/operation/destination bindings, decision and verification generation. It is
inserted before any effective state advance and is unique by receipt/grant/operation.

### 5.10 `DispatchReconciliationV1`

Permanent record of unknown/restore/conflict resolution:

- exact operation/grant/dispatch state and evidence inventory;
- transport quiescence/fencing proof;
- adapter root/epoch/healthy readback generation;
- receipt or permanent no-consumption tombstone reference;
- human/supervisor decision identity where required;
- result state and generation.

It never creates a grant or reactivates restored preparation. Its closed result is
`RECONCILIATION_REQUIRED` while possible/consumed effect status lacks effect evidence,
or `FAILED` only with exact signed no-consumption plus fenced no-inflight proof.

### 5.11 `DispatchEventV1`

One append-only redacted event per transition/decision. Sovereign rows bind exact task,
lease, workload, plan, operation, grant, contract/policy/catalog/capability versions,
decision, latency, event/transition generations and trace identities. Outward
serialization emits only bounded public reason, approved version labels, latency class
and opaque public trace correlation.

## 6. Adapter inbox persistence

The adapter database has a separate application/root identity and no foreign keys into
the coordinator database.

### 6.1 `AdapterStoreMetadataV1`

Singleton schema/root/lifecycle identity, store/inbox/consumption/receipt/conflict/event
generations, ordinary/control queue capacities, current epoch-observer generation and
optional restore/index digests. Lifecycle is `ACTIVE` or `RESTORE_PENDING`.

### 6.2 `GrantInboxEntryV1`

Create-only row keyed by grant ID with unique operation ID and nonce:

- exact canonical grant bytes/digest and verified coordinator key fingerprint;
- destination/protocol/capability bindings;
- independent epoch/root observation;
- state `RECEIVED`, `CONSUMED`, `REFUSED` or `QUARANTINED`;
- receive/current generation and current receipt/event references.

### 6.3 `InboxTransitionV1`

Append-only state transitions:

```text
ABSENT -> RECEIVED
RECEIVED -> CONSUMED
RECEIVED -> REFUSED
RECEIVED -> QUARANTINED
```

No transition returns to `RECEIVED` and no consumed history can be rewritten as refusal.

### 6.4 `AdapterExecutionReceiptV1`

Exact canonical signed receipt and digest, unique receipt/grant/operation, signer
profile/fingerprint, closed decision and created generation. `CONSUMED` receipt and
transition/event commit together before success is returned.

### 6.5 `InboxConflictV1` and `InboxQuarantineV1`

Permanent bounded evidence for reused grant/operation/nonce with conflicting binding,
unknown trusted history, corrupted row, restored old epoch or orphan receipt. It contains
digests/reasons internally and only redacted outward diagnostics. Quarantine never
authorizes consumption.

### 6.6 `AdapterEventV1`

One permanent redacted event per receive, consume, definite refusal, conflict,
quarantine and restore decision, bound internally to task/lease/workload/plan/operation/
grant, contract/protocol/capability versions, decision, latency, generations and trace.

## 7. Closed portable outcomes

### Dispatch request outcomes

- `Dispatched`: exact retained grant identity/digest and redacted state projection.
- `AlreadyDispatched`: same exact retained identity, never newly minted.
- `Denied`: definite pre-transition authority/context/deadline/capacity refusal.
- `Failed`: store/signing failure proved before any dispatch commit.
- `Ambiguous`: uncertain coordinator commit with exact attempt custody and PAUSE.

### Delivery and receipt outcomes

- `Consumed`: exact authentic `CONSUMED` receipt; coordinator state `EXECUTING`.
- `DefinitelyRefused`: exact authentic permanent refusal/no-consumption evidence and
  final coordinator/base state `FAILED` through the normative reconciliation path.
- `Pending`: exact grant remains valid and retained for bounded redelivery/readback.
- `Conflict`: inconsistent permanent evidence; zero state advance and quarantine.
- `OutcomeUnknown`: possible handoff/acceptance cannot be resolved by deadline.
- `ReconciliationRequired`: unknown custody has been opened and no automatic execution
  or retry is possible.

All variants are typed, payload-bounded and redacted in public debug output.

## 8. Atomicity and protocol sequence

### 8.1 Initial coordinator dispatch transaction

```text
guards + dispatch permit held
  -> final durable reload/comparison
  -> comparison record
  -> exact signed grant record
  -> dispatch current record
  -> PREPARING -> DISPATCHING overlay transition
  -> deliverable outbox row
  -> redacted event
  -> all generations
COMMIT
release writer; only then delivery may start
```

Every member commits or none does. Uncertain commit uses exact attempt readback and never
repeats signing blindly.

### 8.2 Adapter receive and consumption

```text
canonical verify + independent epoch observation
  -> transaction A: ABSENT -> RECEIVED + event
reload + second deadline/epoch observation
  -> transaction B: CONSUMED/REFUSED transition + exact signed receipt + event
```

Exact duplicate returns retained state/receipt. Conflicting identity/binding creates
conflict evidence only. The only signed refusal codes after `RECEIVED` are
`GRANT_EXPIRED`, `SUPERVISOR_EPOCH_MISMATCH` and `ADAPTER_PAUSED`; destination,
protocol, capability and inbox-capacity rejection happens before `RECEIVED` and yields
diagnostic/quarantine evidence only.

### 8.3 Coordinator receipt advance

```text
exact grant/state reload
  -> strict receipt verification
  -> receipt row
  -> DISPATCHING -> EXECUTING transition
  -> outbox acknowledged
  -> redacted event
COMMIT
```

No real effect, success, settlement or budget release occurs.

For an exact `REFUSED_DEFINITE` receipt plus fenced no-inflight proof, one V2 transaction
retains the receipt/reconciliation, appends overlay
`DISPATCHING -> OUTCOME_UNKNOWN -> RECONCILIATION_REQUIRED -> FAILED`, appends base
`PREPARING -> FAILED`, releases the exact stored reservation vector once and appends both
event chains. This is the only PLAN-005 budget release.

### 8.4 Ambiguous delivery

Possible handoff triggers bounded exact inbox/receipt readback. Definite absence requires
quiesced/fenced transport, healthy matching adapter root/epoch, deadline closure and a
generation proving no later receive. Each possible-handoff attempt has exactly one
automatic readback sequence: at most four observations, a total 500 ms budget, backoffs
`0/25/75/175 ms` (offsets `0/25/100/275 ms`), and an earlier cut-off at the original
exclusive grant deadline. A retained exact signed receipt remains proof after expiry;
the sequence never renews, extends, replaces or resigns the grant. Exhaustion never
means absence. Otherwise one transaction records
`DISPATCHING -> OUTCOME_UNKNOWN`; explicit reconciliation then records
`OUTCOME_UNKNOWN -> RECONCILIATION_REQUIRED`. A late consumed receipt remains there for
a later effect-aware feature. Only definite no-consumption completes `FAILED`.

## 9. Cross-record and cross-store invariants

1. Every non-`FAILED` dispatch overlay operation points to exactly one invariant-valid
   current PLAN-004 base `PREPARING` operation, attempt, transition, held reservation and
   published recovery record. Overlay `FAILED` instead points to the exact matching base
   `FAILED` transition, released reservation and both failure events.
2. No restored/quarantined/failed/stale PLAN-004 preparation has a new dispatch overlay.
3. Every grant ID, operation ID and nonce occurs at most once in each trust domain.
4. Every `DISPATCHING` record has exact grant, initial transition, outbox and event rows.
5. Every `EXECUTING` record has one authentic `CONSUMED` receipt and transition/event.
6. `OUTCOME_UNKNOWN` and `RECONCILIATION_REQUIRED` never release the PLAN-004 reservation
   or recovery custody; a late consumed receipt never returns them to `EXECUTING`.
7. Adapter `CONSUMED` has exactly one signed consumed receipt and one transition/event.
8. A consumed inbox entry cannot later become refused, missing or reused.
9. Public-key history can verify retained bytes, but private keys are absent from both
   databases and backup artifacts.
10. Cross-store equality is proved by exact grant/receipt/root/epoch/generation digests,
    never by a claimed distributed transaction.
11. Transient inbox absence is not a definite coordinator outcome.
12. Every overlay `FAILED` has the exact signed `REFUSED_DEFINITE` receipt,
    no-consumption tombstone, fenced transport proof, reconciliation record, complete
    normative overlay chain and matching base failure/release transaction.
13. V1 binaries reject V2; V2 ordinary open never performs migration.

## 10. Backup and restore entities

### 10.1 `CoordinatorDispatchBackupManifestV1`

Binds coordinator V2 root/schema/migration receipt, SQLite backup digest, all lifecycle
generations, grant/dispatch/outbox/receipt/reconciliation counts and an exact sorted
grant/receipt inventory digest.

### 10.2 `AdapterInboxBackupManifestV1`

Binds adapter root/schema/epoch observer, SQLite backup digest, lifecycle generations,
inbox/receipt/conflict/quarantine counts and matching inventory digest.

### 10.3 `DispatchBackupIndexV1`

Top-level provisioner-signed manifest-last index:

- backup/restore identity and trusted cut time;
- coordinator and adapter root/manifest/database digests;
- supervisor epoch and complete cross-store inventory digest;
- grant/receipt public verifier profiles, key IDs, fingerprints and revocation metadata;
- PAUSE/quiescence evidence and independent backup order;
- source/toolchain/artifact identity.

No private key or raw secret is a member.

### 10.4 `VerifiedDispatchRestoreV1`

Private maintenance evidence that both empty roots were verified, restored with new
identities/epochs, persisted `RESTORE_PENDING`, started PAUSED, quarantined every old
grant/possible consumption and performed zero redelivery. It is subsystem evidence only
and exposes no activation authority.

## 11. Migration, retention and removal

- V1-to-V2 migration is explicit, paused, backed up and attempt-bound; uncertain commit
  is classified by exact schema/migration-receipt readback.
- V1 remains readable only by V1; V2 retains unchanged base objects. No automatic
  downgrade exists after a grant record.
- Grants, receipts, transitions, conflicts, quarantines and reconciliation tombstones
  are permanent in v1. Automatic pruning and secure-erasure claims are out of scope.
- Removal disables PLAN-005 executable surfaces and reverts source/schema integration in
  an isolated copy, but historical authoritative evidence is preserved or exported for
  audit/reconciliation. PLAN-001 through PLAN-004 protected bytes/behavior remain exact.

## 12. Data classification

| Data | Classification | Retention / exposure |
|---|---|---|
| Canonical grant/receipt bytes | restricted security/operation data | permanent v1 sovereign-store evidence; never public log/Graphify |
| Internal IDs/digests/generations | restricted correlation data | retained with rows; redacted from public output |
| Public verification keys/profiles | internal trust metadata | retained in store/backup for historical verification |
| Private signing keys | secret credential-store data | never stored in PLAN-005 databases, backups, fixtures or evidence |
| Events/metrics | redacted operational metadata | bounded permanent events / declared metric retention |
| Fault fixtures | synthetic public test data | committed only if they contain no secret/private path/content |

Production roots require an approved encrypted-at-rest profile. Process-kill and logical
restore evidence do not prove sector-level deletion or power-loss durability.
